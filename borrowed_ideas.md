# 借鉴记录

> 记录系统从外部项目学习的东西。知道"从哪来的"和知道"是什么"同等重要。

---

## 1. 双时态标注（Graphiti/Zep）
- **借了什么**：`valid_time`（事件发生时间）和 `ingestion_time`（记录入库时间）分离
- **来源**：Graphiti 的双时态边标注——每条关系记录事件时间和摄入时间。Zep 也实现了类似机制
- **采纳时间**：2026-06-12
- **怎么改的**：事实层 Schema v1.1 新增两个时间字段。现有节点的 `time` 字段近似为 `ingestion_time`
- **为什么借**：Claude 指出当前 `time` 字段含义模糊。分开后可以回答"2026-06-11 我们对这件事知道什么"

## 2. ADD-only 模式（Mem0）
- **借了什么**：只追加不修改不删除——Mem0 2026年4月新算法采用 single-pass ADD-only extraction
- **来源**：Mem0 论文 arXiv:2504.19413
- **采纳时间**：2026-06-11（先于 Mem0 论文，但我们一开始就做了）
- **怎么改的**：事实层"进入即锁定"原则。Mem0 后来验证了这个方向
- **为什么借**：独立设计，后来发现与大项目方向一致

## 3. 多信号检索融合（Mem0/Graphiti）
- **借了什么**：语义+关键词+实体多路并行检索，加权合并
- **来源**：Mem0 的 multi-signal retrieval；Graphiti 的分层子图混合查询
- **状态**：已识别，未实施。等系统 300+ 节点时启动
- **怎么改的**：当前检索是串行的（aliases→tags→time）。未来改为并行+置信度加权

## 4. 冲突检测触发生成命题节点（Mem0 反向使用）
- **借了什么**：Mem0 的冲突检测逻辑——发现新事实和旧事实矛盾时触发动作。Mem0 用来自动合并；莱德利基用来生成命题节点
- **状态**：已识别，未实施。需要语义相似度基础设施（Ollama embedding）
- **为什么是反向使用**：Mem0 消除矛盾，我们保留矛盾并给它一个命题节点

## 5. 反向链接 & 孤儿检测（Foam）
- **借了什么**：反向链接面板（查看指向某节点的所有索引条目）+ 孤儿节点检测（找出无入链的遗忘节点）
- **来源**：Foam — VS Code Markdown 知识管理系统，17K stars
- **状态**：已识别，未实施。36 节点不需要，300 节点时启用
- **为什么借**：Foam 是给人用的 PKM，纯 Markdown、本地、无锁定——和莱德利基系统同一哲学。反向链接和孤儿检测是轻量增强，不动架构

## 不能借的（明确拒绝）

| 系统 | 功能 | 拒绝原因 |
|------|------|----------|
| Mem0/Cognee | 冲突合并/更新/覆盖 | 事实层圣域不可修改。它们把记忆当数据库，我们把记忆当历史档案 |
| 所有系统 | 摄入一切，检索时过滤 | 莱德利基的准入规则确保意义密度。不是为了记住更多，是为了记住更重要的 |
| Mem0 | UPDATE/DELETE 操作 | 与圣域根本冲突。撤回不删除只标注 |
